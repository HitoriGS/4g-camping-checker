# Phase 0 技術偵察筆記

實測日期：2026-07-21。用瀏覽器工具實際打開三家電信涵蓋率頁面與 Google 地圖評論，記錄下方結論，作為 Phase 3/4 實作依據。

## 中華電信 (CHT)

- URL: https://coverage.cht.com.tw/coverage/tw.html
- 涵蓋率是疊在 Google Maps 上的彩色 PNG 圖磚（例如 `5G_TW_N1.png`、`5G_DS.png` 等區域圖磚），確認網路請求只有靜態圖磚檔案，**沒有 per-point 查詢 API**。
- 頁面上方有「行動寬頻系統(5G NR+4G LTE)」/「3G基地台關閉進度專區」切換鈕，以及縣市/鄉鎮市區下拉選單 + 地址關鍵字搜尋框。搜尋框只會把地圖 pan/zoom 到該地址，不會回傳文字結果。
- **策略：策略 B（顏色比對）**。用 Playwright 打開頁面、確保選到「5G NR+4G LTE」圖層、用搜尋框定位到目標地址（或直接操作地圖 API 移動到座標）、等待圖磚載入完成（`networkidle` + 額外等待），對目標座標在螢幕上的像素位置截圖取樣。
- 圖例色票（見頁面下方表格）用 `scripts/calibrate-colors.ts` 之後手動執行擷取，目前已知圖例分為「5G/4G 室外優/良」「5G/4G 室外可/弱」等分級，各自對應不同深淺的藍/綠色塊。

## 台灣大哥大 (TWM)

- URL: https://www.taiwanmobile.com/mobile/calculate/cover_map.html
- 介面是「選擇縣市 → 選擇鄉鎮市區 → 輸入城鎮/區域/街道名稱」的下拉選單/文字框表單（非自由地圖拖曳搜尋），送出條件後頁面下方會渲染出彩色地圖疊加層（實測畫面確認為類似 CHT 的色塊覆蓋，非純文字結果）。
- 下拉選單為自訂 combobox（非原生 `<select>`），需要用點擊展開再選項目的方式操作，無法用 `form_input` 直接設值。
- **策略：策略 B（顏色比對）**，流程與 CHT 類似，但需要先將 Google geocode 回傳的地址元件轉換成台灣大哥大下拉選單使用的縣市/鄉鎮市區文字（可能有用字差異，例如「台」vs「臺」，需要 `config/taiwanRegions.ts` 做正規化對照表）。

## 遠傳電信 (FET)

- URL: https://ecare.fetnet.net/DigService/serviceCoverageController/serviceCoveragePage
- 介面同樣是「4.5G+5G網路涵蓋率類型 → 選擇縣市 → 選擇區域 → 輸入路名 → 查詢」表單，送出後會渲染一個「地圖」region 顯示結果。
- 因表單自動化流程與 TWM 高度相似，且頁面同樣有 `地圖` 區塊，初步判斷也是圖形疊加而非文字結果，但**尚未完整跑完一次查詢確認渲染結果**，建議在 Phase 4 實作時第一步就先手動跑一次完整查詢並截圖確認，若結果其實是文字（例如百分比或等級字串），優先改用策略 A（文字擷取），比顏色比對更準。

## Google Maps 評論

- 直接用 Google 搜尋/地圖找到露營區地點頁面後，頁面內建「評論摘要」區塊，除了整體星等分布外，會顯示**自動生成的主題關鍵字標籤**（例如某露營區顯示「風景 37」「兔子 34」「廁所 20」「清境 18」+6 個更多標籤），每個標籤可點擊篩選出提到該主題的評論。這是抓取「網路/訊號」相關評論的第一個可用管道：**如果某露營區的訊號問題夠多人在評論中提到，很可能會被 Google 自動歸類成一個主題標籤**（例如「訊號」「網路」），可以直接抓標籤清單比對關鍵字，命中就點擊該標籤取得篩選後的評論列表；沒有命中不代表沒人提過，只代表沒有形成夠強的主題，此時要 fallback 到方法二。
- 每則評論卡片含有：評論者名稱、在地嚮導/評論數/相片數、星等（圖示，需用 aria-label 或 title 屬性讀取，而非顏色）、相對時間文字（例如「4 個月前」「3 年前」）、評論全文（超長會被截斷並顯示「更多」按鈕）。
- 頁面有「更多評論 (341)」按鈕可載入更多評論（分頁或無限捲動），可作為方法二：逐頁載入全部評論後，在抓到的文字中用關鍵字（訊號/網路/收訊/4G/5G/斷線/收不到）做 client-side 篩選，此法更全面但較慢、要處理捲動載入與大量 DOM。
- 使用者原本設想的「放大鏡關鍵字搜尋評論」輸入框，在本次測試環境（透過 `google.com` 導向的內嵌地圖檢視）沒有成功渲染出來（可能是這個沙盒瀏覽器環境對完整版 `maps.google.com` 應用的相容性問題，導致停留在簡化版介面）。**建議 Phase 3 實作時用正式的 Playwright（完整桌面版 User-Agent、直接導到 `https://www.google.com/maps/place/...` 完整網址）重新確認該搜尋框是否存在**；若存在則優先用它（最精準、最少 DOM 要處理），若經確認不存在或不穩定，改用上述「主題標籤 + 分頁載入全部評論後關鍵字篩選」的組合方案，兩者本來就設計成互為備援。
- 不需要登入即可瀏覽以上內容（測試過程全程未登入 Google 帳號）。

## 結論與 Phase 4 待辦

1. CHT、TWM 確認為策略 B（顏色比對）；FET 待 Phase 4 一開始手動跑一次確認是文字還是圖形。
2. Google 評論擷取採雙軌：主題標籤優先命中關鍵字 → 若無命中或標籤太少，fallback 分頁載入 + 全文關鍵字篩選；若正式 Playwright 環境下確認有「評論內搜尋」輸入框，則優先用它取代上述兩法（更準更快）。
3. 評論星等建議用 DOM 的 `aria-label`（通常包含「評分為 5 顆星，最高 5 顆星」之類完整文字）解析數字，不要用圖片顏色判斷。
